## 1. ЗАГАЛЬНІ ВІДОМОСТІ

### 1.1 Загальна характеристика

Геоінформаційна система (ГІС) призначена для комплексного багатофакторного аналізу рекреаційного потенціалу територій України з метою визначення оптимальних локацій для розміщення об'єктів рекреаційної інфраструктури.

### 1.2 Сфера застосування

- Регіональне планування рекреаційних зон
- Оцінка інвестиційної привабливості територій
- Профілактика лісових пожеж через створення облаштованих місць відпочинку
- Екологічне планування поблизу об'єктів природно-заповідного фонду
- Розробка стратегій розвитку туристичної інфраструктури

### 1.3 Цільова аудиторія

- Науковці у сфері геоінформатики та рекреаційної географії
- Органи регіонального та місцевого самоврядування
- Інвестори та девелопери туристичної інфраструктури
- Екологічні організації та природоохоронні установи
- Фахівці з просторового планування

## 2. ПРИЗНАЧЕННЯ СИСТЕМИ

### 2.1 Основні функції

**Аналітичні функції:**
- Багатофакторна оцінка рекреаційного потенціалу 24 областей України
- Розрахунок інтегрального показника пріоритетності територій (0-90 балів)
- Ідентифікація оптимальних локацій для розміщення рекреаційних об'єктів
- Кластерний аналіз людських лісових пожеж
- Оцінка природно-заповідного фонду як атрактора

**Візуалізаційні функції:**
- Інтерактивна картографічна візуалізація даних
- Багатошарова система відображення геопросторових об'єктів
- Динамічні popup-вікна з деталізованою інформацією
- Кольорове кодування за рівнем пріоритетності

**Функції експорту:**
- Генерація детальних HTML-звітів з повним розрахунком факторів
- Експорт даних у форматі JSON для подальшої обробки
- Збереження результатів аналізу для архівування

### 2.2 Наукова цінність

Система реалізує міжнародно визнану методологію Analytic Hierarchy Process (AHP) Томаса Сааті (1980) для зважування критеріїв оцінки. Consistency Ratio системи становить 0.16%, що значно нижче критичного порогу 10% та свідчить про відмінну узгодженість ваг факторів.

## 3. АРХІТЕКТУРА ТА ТЕХНОЛОГІЇ

### 3.1 Архітектурна модель

Система реалізована за архітектурним патерном Client-Server з наступними компонентами:

**Frontend (Клієнтська частина):**
- Фреймворк: React 18.x
- UI-бібліотека: Tailwind CSS 3.x
- Картографія: Leaflet 1.9.x
- Відображення даних: GeoJSON
- Інтерактивність: React Hooks, Context API

**Backend (Серверна частина):**
- Фреймворк: FastAPI (Python 3.11+)
- Обчислювальний движок: NumPy, Pandas
- Геопросторова обробка: Geopy, Shapely
- Валідація даних: Pydantic
- API архітектура: RESTful

**База даних:**
- СУБД: MongoDB 7.x
- Тип: NoSQL документо-орієнтована
- Структура даних: GeoJSON-сумісні колекції

### 3.2 Структура даних

**Колекції MongoDB:**

1. `regions` - Адміністративні області України
   - Назва області
   - Геометрія (полігони)
   - Демографічні показники
   - Економічні індикатори

2. `pfz_objects` - Об'єкти природно-заповідного фонду
   - Категорія (НПП, РЛП, заказник, пам'ятка природи)
   - Площа (га)
   - Координати
   - Рік створення

3. `forest_fires` - Лісові пожежі
   - Координати
   - Причина (людський фактор / природна)
   - Дата
   - Область

4. `recreational_points` - Існуючі рекреаційні об'єкти
   - Тип (готель, мотель, кемпінг)
   - Координати
   - Область

5. `roads` - Транспортна мережа
   - Тип дороги (міжнародна, національна, регіональна)
   - Геометрія (лінії)
   - Ідентифікатор траси

### 3.3 API Endpoints

**Основні маршрути:**


## 4. НАУКОВО-МЕТОДОЛОГІЧНА БАЗА

### 4.1 Метод аналітичної ієрархії (AHP)

Система використовує AHP для визначення відносної важливості кожного з 7 факторів оцінки. Метод базується на парному порівнянні критеріїв та побудові матриці переваг.

**Матриця парних порівнянь:**

|          | Попит | ПЗФ | Природа | Транспорт | Інфра | Пожежі | Насич |
|----------|-------|-----|---------|-----------|-------|--------|-------|
| Попит    | 1     | 2   | 3       | 3         | 4     | 5      | 3     |
| ПЗФ      | 0.5   | 1   | 2       | 2         | 3     | 4      | 2     |
| Природа  | 0.33  | 0.5 | 1       | 1         | 2     | 3      | 1     |
| Транспорт| 0.33  | 0.5 | 1       | 1         | 2     | 3      | 1     |
| Інфра    | 0.25  | 0.33| 0.5     | 0.5       | 1     | 2      | 0.5   |
| Пожежі   | 0.2   | 0.25| 0.33    | 0.33      | 0.5   | 1      | 0.33  |
| Насич    | 0.33  | 0.5 | 1       | 1         | 2     | 3      | 1     |

**Нормалізовані ваги:**
- Фактор 1 (Попит): 25% (25 балів)
- Фактор 2 (ПЗФ атрактор): 20% (20 балів)
- Фактор 3 (Природа): 15% (15 балів)
- Фактор 4 (Транспорт): 15% (15 балів)
- Фактор 5 (Інфраструктура): 10% (10 балів)
- Фактор 6 (Пожежі): 5% (5 балів)
- Фактор 7 (Насиченість): 15% (-15 балів штраф)

**Перевірка узгодженості:**
- λmax = 7.0112
- CI (Consistency Index) = 0.0019
- RI (Random Index) = 1.32
- CR (Consistency Ratio) = 0.0016 = 0.16%

Оскільки CR < 10%, матриця вважається узгодженою.

### 4.2 Формула розрахунку пріоритету

**Для областей:**

```
Pregion = F1 + F2 + F3 + F4 + F5 + F6 + F7
```

де:
- F1: Попит від населення (0-25)
- F2: ПЗФ як атрактор (0-20)
- F3: Природні умови (0-15)
- F4: Транспортна доступність (0-15)
- F5: Інфраструктура (0-10)
- F6: Профілактика пожеж (0-5)
- F7: Насиченість (0 до -15)

**Для зон будівництва:**

```
Pzone = Попит(25) + Атрактор(20) + Природа(15) + Транспорт(15) + 
        Інфраструктура(10) + Пожежі(5) - Насиченість(15)
```

Максимум: 90 балів  
Мінімум: 75 балів (з врахуванням максимального штрафу)

### 4.3 Алгоритм ідентифікації зон

Система визначає рекомендовані локації у 3 категоріях:

**1. Кластери людських лісових пожеж**

Алгоритм:
```python
for each fire in human_fires:
    nearby_fires = count_fires_within_radius(fire.coords, 10km)
    if nearby_fires >= 3:
        cluster_center = calculate_centroid(nearby_fires)
        zones.append(cluster_center)
```

Наукове обґрунтування: 80% рекреаційних пожеж відбуваються поза офіційними місцями (NW Fire Science Consortium, 2020). Облаштовані місця знижують ризик на 40%.

**2. Зони біля об'єктів ПЗФ**

Алгоритм:
```python
top_pfz = select_top_2_by_visitors(region.pfz_objects)
for pfz in top_pfz:
    distance = random(3, 8)  # км від центру
    angle = random(0, 360)
    zone = calculate_offset_coords(pfz.center, distance, angle)
    zones.append(zone)
```

Відстань 3-8 км забезпечує баланс між близькістю до атрактора та мінімізацією екологічного впливу.

**3. Зони вздовж міжнародних трас**

Алгоритм:
```python
for road in international_roads:
    points = generate_points_along_road(road, interval=50-100km)
    zones.extend(points)
```

Інтервал 50-100 км відповідає середній відстані транзитної зупинки для відпочинку водіїв.

### 4.4 Радіус зони будівництва

Кожна рекомендована точка має радіус впливу **±5 км**. У межах цього радіусу:

1. Здійснюється перевірка конкуренції (існуючі рекреаційні об'єкти)
2. Можна обирати конкретне місце для будівництва
3. Враховуються локальні умови (доступність землі, комунікації)

Формула штрафу за насиченість:
```python
competitors = count_recreational_points_within(zone.coords, 5km)

if competitors == 0:
    penalty = 0
elif competitors <= 3:
    penalty = -5
elif competitors <= 7:
    penalty = -10
elif competitors <= 12:
    penalty = -12
else:
    penalty = -15
```

**Крок 2: Запуск аналізу**

1. Натисніть кнопку «Аналізувати область»
2. Система відправить GET запит до `/api/analyze/{region_name}`
3. З'явиться індикатор завантаження «Аналіз...»
4. Після завершення (1-3 секунди) відобразяться результати

**Крок 3: Перегляд результатів**

Ліва панель відобразить:

1. **Загальний бал області** (0-90)
   - Кольорове кодування:
     - 75-90: зелений (високий пріоритет)
     - 60-74: помаранчевий (середній)
     - <60: червоний (низький)

2. **Оцінка за факторами:**
   - Попит: X/25
   - ПЗФ атрактор: X/20
   - Природа: X/15
   - Транспорт: X/15
   - Інфраструктура: X/10
   - Пожежі: +X/5
   - Насиченість: -X/15

3. **Деталізовані аккордеони:**
   Натисніть на кожен фактор для перегляду:
   - Методологія розрахунку
   - Вихідні дані
   - Проміжні обчислення
   - Візуалізація показників

**Крок 4: Перегляд рекомендованих зон**

1. Після аналізу на карті з'являться червоні пунктирні кола
2. Натисніть на будь-який маркер для перегляду popup
3. Popup містить:
   - Назва зони (наприклад, "Кластер пожеж #1")
   - Тип (fire_prevention / near_pfz / roadside)
   - Пріоритет X/90
   - 3 ключові обґрунтування
   - Рекомендована інфраструктура
   - Юридичний статус

### 5.4 Інтерпретація маркерів зон

**Типи зон:**

1. **Кластер пожеж #N** (type: fire_prevention)

   - Обґрунтування:

   - Інфраструктура:

     - Майданчик для пікніка з лавками
     - Контейнери для сміття
     - Інформаційні стенди про пожежну безпеку

2. **Біля [Назва ПЗФ]** (type: near_pfz)
   - Іконка: червоне коло
   - Додатково: відстань до ПЗФ (X км)
   - Обґрунтування:
     - [Назва НПП] - 30,000 відвідувачів/рік
     - Попит: X балів, Пожежі: Y балів
     - Конкуренція: N р.п.
   - Інфраструктура:
     - Екологічний готель / глемпінг
     - Інформаційний центр для відвідувачів
     - Екологічні стежки з гідами
     - Станція оренди обладнання

3. **Вздовж [Назва траси]** (type: roadside)
   - Іконка: червоне коло
   - Обґрунтування:
     - [М06 Київ-Чоп] - головна траса (5,000+ авто/день)
     - Транспорт: X балів, Попит: Y балів
     - Конкуренція: N р.п. на маршруті
   - Інфраструктура:
     - Мотель або придорожній готель
     - Безпечна стоянка з охороною
     - Кафе та міні-маркет
     - Санітарні приміщення

**Кольорове кодування пріоритету:**
- 85-90 балів: яскраво-зелений (найвищий)
- 75-84 бали: помаранчевий (високий)
- 60-74 бали: жовтий (середній)

## 6. АНАЛІЗ РЕЗУЛЬТАТІВ

### 6.1 Інтерпретація балів області

**Високий пріоритет (75-90 балів):**
- Область має виражений потенціал для розвитку рекреації
- Рекомендується до інвестування
- Наявні атрактори (НПП, висока природна цінність)
- Доступна транспортна інфраструктура

**Середній пріоритет (60-74 бали):**
- Область має помірний потенціал
- Потребує додаткового аналізу конкретних локацій
- Можливі обмеження у транспортній доступності або інфраструктурі

**Низький пріоритет (<60 балів):**
- Область не є пріоритетною для рекреаційних інвестицій
- Можливі причини:
  - Низький попит (мале населення)
  - Відсутність значних ПЗФ об'єктів
  - Слабка транспортна мережа
  - Висока насиченість існуючими об'єктами

### 6.2 Аналіз факторів

**Фактор 1: Попит (0-25 балів)**

Розрахунок:
```
Річний попит = Населення × 8 візитів/рік
Існуюча пропозиція = Кількість р.п. × 300 осіб/рік
Дефіцит = Попит - Пропозиція
Бал = normalize(Дефіцит, 0, 25)
```

Високий бал (>20): значний незадоволений попит, відмінні перспективи  
Середній бал (15-20): помірний попит, стабільний ринок  
Низький бал (<15): насичений ринок або низький попит

**Фактор 2: ПЗФ атрактор (0-20 балів)**

Розрахунок:
```
Зважена сума ПЗФ:
- НПП: ×5
- РЛП: ×3
- Заказник: ×1.5
- Пам'ятка природи: ×1

Коригування на площу області
Бал = normalize(Зважена сума, 0, 20)
```

Високий бал (>15): наявні великі НПП з високою відвідуваністю  
Середній бал (10-15): є РЛП або кілька заказників  
Низький бал (<10): мало об'єктів ПЗФ

**Фактор 3: Природні умови (0-15 балів)**

Дві компоненти:

A) Лісове покриття (0-11):
```
Лісистість < 10%: 3 бали
Лісистість 10-20%: 6 балів
Лісистість 20-30%: 9 балів
Лісистість > 30%: 11 балів
```

B) Водні об'єкти (0-4):
```
Є великі водойми: +4 бали
Немає: 0 балів
```

Високий бал (>12): ідеальні природні умови для рекреації  
Середній бал (8-12): достатні природні ресурси  
Низький бал (<8): обмежені природні активи

**Фактор 4: Транспортна доступність (0-15 балів)**

Розрахунок:
```
Міжнародні траси: +15 балів
Національні траси: +12 балів
Регіональні траси: +8 балів
Місцеві дороги: +5 балів
```

Високий бал (>12): відмінна транспортна доступність  
Середній бал (8-12): прийнятна доступність  
Низький бал (<8): слабка дорожня мережа

**Фактор 5: Інфраструктура (0-10 балів)**

Розрахунок:
```
Готелі у столиці області:
  > 50: 10 балів
  30-50: 8 балів
  10-30: 5 балів
  < 10: 3 бали
```

Високий бал (>8): розвинена туристична інфраструктура  
Середній бал (5-8): базова інфраструктура  
Низький бал (<5): слабка інфраструктура

**Фактор 6: Профілактика пожеж (0-5 балів)**

Бонус за наявність людських пожеж:
```
10+ пожеж: +5 балів (критична потреба)
5-9 пожеж: +3 бали (висока потреба)
1-4 пожежі: +2 бали (помірна потреба)
0 пожеж: 0 балів
```

Парадокс: БІЛЬШЕ пожеж = ВИЩИЙ бал, оскільки це індикатор потреби в облаштованих місцях.

**Фактор 7: Насиченість (0 до -15 балів)**

Штраф за конкуренцію:
```
0 конкурентів у радіусі 5 км: 0 балів
1-3 конкурента: -5 балів
4-7 конкурентів: -10 балів
8-12 конкурентів: -12 балів
13+ конкурентів: -15 балів
```

Мінімальний штраф: низька конкуренція, вільний ринок  
Максимальний штраф: перенасичена територія

## 8. ІМПОРТ КОРИСТУВАЦЬКИХ ДАНИХ

### 8.1 Загальні вимоги

**Підтримувані формати:**
- GeoJSON (RFC 7946)
- Кодування: UTF-8
- Координатна система: WGS84 (EPSG:4326)

**Обмеження:**
- Максимальний розмір файлу: 10 MB
- Максимальна кількість об'єктів: 10,000
- Валідна GeoJSON структура

### 8.2 Інтерфейс імпорту

**Доступ:**
1. Верхнє меню → «Імпорт даних»
2. Або: Головна сторінка → «Імпорт даних» у footer

**Структура сторінки:**
- Інструкції з імпорту
- 5 типів даних з окремими формами
- Приклади GeoJSON для кожного типу
- Кнопка «Завантажити приклади» (ZIP архів)

### 8.3 Типи даних для імпорту

#### 8.3.1 Області (regions)

**Призначення:** Додати нові адміністративні одиниці

**Структура GeoJSON:**
```json
{
  "type": "FeatureCollection",
  "features": [
    {
      "type": "Feature",
      "geometry": {
        "type": "Polygon",
        "coordinates": [[[30.5, 50.4], [30.6, 50.4], ...]]
      },
      "properties": {
        "name": "Нова область",
        "population": 1000000,
        "area_km2": 5000
      }
    }
  ]
}
```

**Обов'язкові поля:**
- `name` (string): назва області
- `population` (number): населення
- `geometry`: Polygon або MultiPolygon

**Опціональні поля:**
- `area_km2` (number): площа у км²

#### 8.3.2 Об'єкти ПЗФ (pfz_objects)

**Призначення:** Додати нові природно-заповідні об'єкти

```json
{
  "type": "FeatureCollection",

    {
      "type": "Feature",

        "type": "Point",

        "area_ha": 50000,

      }
    }
  ]
}
```

**Обов'язкові поля:**
- `name` (string): назва об'єкта
- `category` (string): НПП | РЛП | заказник | пам'ятка природи
- `region` (string): область
- `geometry`: Point

**Опціональні поля:**
- `area_ha` (number): площа у гектарах
- `year_created` (number): рік створення

### 8.4 Процес імпорту

**Покрокова інструкція:**

1. **Підготовка файлу**
   - Створіть GeoJSON згідно зі специфікацією вище
   - Перевірте валідність на https://geojson.io

   - Перейдіть на сторінку «Імпорт даних»

   - Система автоматично перевірить:
     - Формат GeoJSON
     - Наявність обов'язкових полів
     - Валідність геометрії

     - "Успішно завантажено N об'єктів"
     - Зелена галочка
   - Дані одразу доступні для аналізу

5. **Перевірка**
   - Поверніться до карти
   - Увімкніть відповідний шар
   - Перевірте відображення нових об'єктів

6. Імпортуйте через інтерфейс

## 9. ІНТЕРПРЕТАЦІЯ РЕЗУЛЬТАТІВ

### 9.1 Використання результатів для прийняття рішень

**Для інвесторів:**

1. **Вибір області для інвестування**
   - Пріоритет 1: Області з балом 75-90
   - Оцініть фактор попиту (>20 балів)
   - Перевірте наявність потужних атракторів (ПЗФ >15)

2. **Вибір конкретної локації**
   - Оберіть зону з пріоритетом 85-90
   - Перевірте тип зони:
     - Кластери пожеж: швидка віддача, соціальна цінність
     - Біля НПП: стабільний потік туристів
     - Вздовж трас: транзитні клієнти
   - Оцініть конкуренцію (штраф насиченості)

3. **Планування інфраструктури**
   - Використайте рекомендовану інфраструктуру з popup
   - Враховуйте специфіку локації
   - Оцініть інвестиційні витрати

**Для органів влади:**

1. **Регіональне планування**
   - Ідентифікація пріоритетних територій
   - Розподіл бюджетних коштів
   - Залучення приватних інвестицій

2. **Профілактика пожеж**
   - Пріоритизація кластерів з високою кількістю інцидентів
   - Планування облаштованих місць відпочинку
   - Інформаційні кампанії

3. **Розвиток туризму**
   - Маркетинг областей з високим потенціалом
   - Розробка туристичних маршрутів
   - Інтеграція з ПЗФ об'єктами

**Для екологічних організацій:**

1. **Зниження антропогенного впливу**
   - Облаштування місць біля НПП зменшує стихійний туризм
   - Контрольовані зони замість безконтрольного розпилення відвідувачів

2. **Моніторинг пожежної небезпеки**
   - Кластерний аналіз для виявлення проблемних зон
   - Профілактичні заходи у критичних локаціях

### 9.2 Обмеження системи

**Методологічні обмеження:**

1. **Статичність даних**
   - Система використовує історичні дані
   - Не враховує майбутні зміни (нова інфраструктура, демографічні зміни)
   - Рекомендовано оновлювати дані щороку

2. **Спрощення реальності**
   - Радіус зони будівництва (5 км) - це приблизна оцінка
   - Не враховуються локальні особливості (рельєф, власність землі)
   - Потрібна додаткова експертиза на місці

3. **Відсутність економічного моделювання**
   - Система не розраховує ROI (Return on Investment)
   - Не оцінює строк окупності
   - Не враховує вартість землі та будівництва

4. **Обмеженість факторів**
   - 7 факторів не вичерпують всіх аспектів
   - Відсутні: кліматичні умови, сезонність, політична стабільність

**Технічні обмеження:**

1. **Точність геолокації**
   - Координати об'єктів можуть мати похибку ±100м
   - Залежить від якості вихідних даних

2. **Продуктивність**
   - Аналіз великих областей може займати 2-5 секунд
   - Візуалізація 1000+ об'єктів на карті може уповільнити інтерфейс

3. **Сумісність браузерів**
   - Оптимізовано для Chrome, Firefox, Edge (останні версії)
   - Safari може мати обмеження у роботі з GeoJSON

**Рекомендації щодо використання:**

1. Використовуйте систему як інструмент попередньої оцінки
2. Обов'язково проводьте польову експертизу обраних локацій
3. Доповнюйте аналіз економічним моделюванням
4. Консультуйтеся з місцевими експертами
5. Враховуйте юридичні аспекти (власність землі, зонування)

### 10.2 Часті питання (FAQ)

**Q: Чому моя область має низький бал?**  
A: Низький бал може бути через: низький попит (мале населення), відсутність значних ПЗФ об'єктів, слабку транспортну мережу, або високу насиченість існуючими об'єктами. Перевірте деталізовані аккордеони для кожного фактора.

**Q: Чи можу я змінити ваги факторів?**  
A: Поточна версія системи використовує фіксовані ваги згідно з AHP методологією. Для зміни ваг потрібна модифікація бекенду.

**Q: Як інтерпретувати радіус зони будівництва 5 км?**  
A: Це означає, що в межах 5 км від центральної точки маркера можна шукати конкретне місце для будівництва. Враховуйте локальні умови (доступність землі, комунікації, рельєф).

**Q: Чому кластери пожеж мають високий пріоритет?**  
A: Це парадоксальна логіка: більше пожеж = вища потреба в облаштованих місцях. Згідно з дослідженнями, 80% рекреаційних пожеж відбуваються поза офіційними місцями. Облаштовані пункти знижують ризик на 40%.

**Q: Як додати власні дані?**  
A: Використовуйте розділ "Імпорт даних". Підготуйте GeoJSON файл згідно зі специфікацією (розділ 8), завантажте через інтерфейс.

**Q: Чому не відображаються мої імпортовані дані?**  
A: Перевірте: 1) успішність завантаження (зелена галочка), 2) увімкнений відповідний шар на карті, 3) область співпадає з вашим вибором у dropdown, 4) валідність GeoJSON структури.

**Q: Як експортувати результати для презентації?**  
A: Використовуйте "Експорт HTML", потім збережіть як PDF через Ctrl+P. Для обробки даних використовуйте "Експорт JSON".

**Q: Чи враховує система економічні показники?**  
A: Система враховує попит (демографічний фактор) та інфраструктуру, але не розраховує ROI чи строк окупності. Для фінансового моделювання використовуйте додаткові інструменти.

**Q: Як часто потрібно оновлювати дані?**  
A: Рекомендується оновлювати дані щороку, особливо: статистику пожеж, дані про нові рекреаційні об'єкти, демографічні показники.

## ДОДАТОК А. ГЛОСАРІЙ ТЕРМІНІВ

**AHP (Analytic Hierarchy Process)** - Метод аналітичної ієрархії, розроблений Томасом Сааті для багатокритеріального прийняття рішень.

**Consistency Ratio (CR)** - Коефіцієнт узгодженості матриці парних порівнянь. CR < 10% вважається прийнятним.

**GeoJSON** - Формат обміну геопросторовими даними на базі JSON (RFC 7946).

**НПП** - Національний природний парк. Категорія ПЗФ найвищого рівня охорони.

**ПЗФ** - Природно-заповідний фонд України. Система територій та об'єктів природно-заповідного фонду.

**РЛП** - Регіональний ландшафтний парк. Категорія ПЗФ регіонального значення.

**Рекреаційний потенціал** - Сукупність природних, культурно-історичних та соціально-економічних передумов для організації рекреаційної діяльності.

**Насиченість території** - Щільність існуючих рекреаційних об'єктів на одиницю площі або населення.

**Кластер пожеж** - Географічне скупчення 3+ лісових пожеж в радіусі 10 км.

**WGS84** - Всесвітня геодезична система координат 1984 року (EPSG:4326).
